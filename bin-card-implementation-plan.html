Bin Card Feature — Implementation Plan

A per-SKU, date-range stock movement report for a distributor — matching your target screenshots, on the existing SFA stock ledger.

✓ Your decisions applied

  • Stock Taking + Stock Adjustment → one column ("Stock Adjustment"), sourced from the StockTakingAdjustment movement (signed: + for gains, − for losses).
  • "Dis Return Qty" column removed.
  • "Current Stock (Pcs)" = latest physical count (most recent stock-take count), so Stock Variance compares book stock vs counted stock.

// Rep Return D/E excluded · Dis Return removed · verified vs screenshots
Reconciliation guarantee: because every column except Rep Return D/E is summed from the same ledger that maintains the live balance, End Stock will always equal the ledger's true closing balance. The report can compute End Stock either way and assert they match — a built-in correctness check.

2 Backend — API sfa_api

One new endpoint on the existing Stock vertical slice. No schema changes, no migration — the ledger and indexes already exist (composite index (DistributorId, ProductId, TransactedAt) covers the range scan).

Features/Stock/ ├── DTOs/ │ ├── BinCardRowDto.cs # one record per SKU (all 16 fields) │ └── BinCardResponseDto.cs # header (distributor, range) + rows + grand totals ├── Requests/ │ └── BinCardQuery.cs # distributorId, from, to (validated) ├── Repositories/ │ ├── IStockRepository.cs # + GetBinCardAggregatesAsync(...) │ └── StockRepository.cs # grouped SUM queries (movements, opening, latest count) ├── Services/ │ ├── IBinCardService.cs │ └── BinCardService.cs # assembles columns, computes End/Value/Variance ├── Validators/ │ └── BinCardQueryValidator.cs # to >= from, range cap (e.g. 1 year) ├── Controllers/ │ └── StockController.cs # + GET .../distributors/{id}/bin-card └── StockServiceExtensions.cs # register IBinCardService

Endpoint

GET /api/v1/stock/distributors/{distributorId}/bin-card?from=2026-06-01&to=2026-06-15 → ApiResponse<BinCardResponseDto> { distributorName, from, to, rows[], totals }

Query strategy (set-based, no N+1)

  • Movements in range: one GROUP BY ProductId, TransactionType, StockType, Direction SUM over StockTransactions where date ∈ [from, to] → fills Invoice/Resaleable/Deleted/Adjustment/Sold/FreeIssues columns.
  • Opening stock: per (product, pool), the QuantityAfter of the latest movement TransactedAt < from → summed per product. (Window function / lateral, one round-trip.)
  • Rep Return D/E: one grouped SUM over BillingItems joined to Billings (distributor + date + returnType ∈ Damage/Expire).
  • Latest physical count: per product, CountedQuantity from the most recent StockTakingSubmission for the distributor.
  • Service joins these by ProductId in memory, computes End Stock / Closing Value / Variance, returns rows + totals. ~4 grouped queries total regardless of SKU count.
Date semantics: from/to are inclusive business dates. Open Stock = balance at from 00:00; in-range sums cover from 00:00 → to 23:59:59; End Stock = balance at end of to. Boundaries converted from the business timezone to the UTC TransactedAt stored on each row.
