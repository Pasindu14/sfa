Production Robustness Review · SFA API

Duplicate-Request & Industry-Grade Resilience Audit

A deep read of the sfa_api codebase across the five highest-risk domains for a Sales Force Automation platform: duplicate requests, concurrency & races, transaction atomicity, authorization / data isolation, and data integrity.

Scope: D:\Github\sfa\sfa_api Stack: .NET 8 · EF Core · PostgreSQL Method: 5 parallel full-file code audits Date: 2026-06-22

The honest bottom line

What is safe: The two failures that would corrupt a real distributor's books — overselling stock and colliding invoice numbers — are correctly prevented by row-level SELECT … FOR UPDATE locks and PostgreSQL nextval sequences. The order → stock → ledger → audit pipeline is genuinely all-or-nothing. Auth token handling is strong.

What was NOT safe (at audit time — now remediated; see status banner below): The specific thing asked about — duplicate request handling — has a real gap. The idempotency layer is a post-completion cache with no in-flight lock, Billing has no database-level duplicate backstop, and the web client sends no idempotency key at all. Several read endpoints also have IDOR (cross-territory data access) gaps. These are fixable and none silently corrupts money totals — but they are genuine and should be closed before scale.

✅ Remediation status — all 24 findings addressed (2026-06-22)

Every finding below has since been fixed or reviewed & accepted with a documented decision. Full .NET suite green: 873 unit + 500 integration (1 known skip).

Critical (5): all fixed — Billing gained a DB-level ClientBillId unique backstop + idempotent fast-path; the idempotency middleware gained an in-flight lock + upsert; the DailyRouteAssignment delete fall-through, SalesInvoices and GRNs IDOR are closed.  High (6): all fixed — Billings/Users read scoping, soft-delete global query filters, idempotency key scoped by method+path, and the web client now sends keys on every mutation.  Medium (7): 2 fixed (billing-setter locks, Outlets territory scoping), 5 reviewed & accepted/documented.  Low (6): 4 fixed (PO sequence-gap, anonymous-key skip, validator upper bounds, global fallback auth policy + Swagger prod-gate), 2 accepted/documented.

Critical findings

Real paths to duplicate writes or cross-tenant data access. These are the ones to close first.

Critical · Duplicates

1. Idempotency has no in-flight lock — concurrent retries both get through

WhereCommon/Middleware/IdempotencyMiddleware.cs:38-79 · PostgresIdempotencyService.cs:13-39

The middleware reads the cache, then only writes it after the controller fully completes with a 2xx. There is no lock and no "in-progress" marker. Two identical requests arriving before the first finishes both see an empty cache, both run the full controller, and both create separate records.

SFA risk YES — but narrowed by the mobile client Real in the slow-bill / lost-response timeout window only.
Mobile mitigation (verified): the Flutter app generates the idempotency key (clientBillId) once via uuid.v4() at save (create_bill_bloc.dart:271), persists it to SQLite, and reuses it verbatim on every retry. It also disables the save button while submitting (cart_list.dart:499) and has no Dio auto-retry. So casual double-taps and sequential retry-after-failure are already handled — the server cache dedupes those.
Residual biscuit-floor window: Dio's receive timeout is 15 s (dio_client.dart:21). On a heavy bill the device can abandon an in-flight request at 15 s and re-send the same key while the original is still committing server-side → two same-key requests overlap → the lockless middleware lets the second past the still-empty cache → potential second bill. The device's _inFlight set can't help because it already gave up on the first request. Fix belongs on the server, not the app.
Critical · Duplicates

2. Billing create has no database-level duplicate backstop

WhereFeatures/Billings/Services/BillingService.cs:226-340 · AppDbContext.cs:749-847

The advisory lock is keyed billing:create:{salesRepId} and is fail-fast — it serializes a rep's requests but does not deduplicate them. The client's clientBillId is sent only as the idempotency header and is never persisted, so the database cannot reject a duplicate. If finding #1 lets two requests through, nothing downstream stops two bills. Contrast NotBilling, which does have a filtered unique index on (SalesRepId, OutletId, NotBillingDate) — the correct pattern, just not applied to Billing.

SFA risk YES — real Billing is the highest-value duplicate target and depends entirely on the racy middleware cache.
Proof the fix belongs here: the mobile not-billing flow uses identical client code (same once-per-record key, same 15 s timeout, same outbox) yet cannot duplicate — because NotBilling has a unique index on (SalesRepId, OutletId, NotBillingDate) (AppDbContext.cs:896-901). Billing has no equivalent. Mirroring that — persist clientBillId on the Billing row + unique index — closes finding #1's residual window regardless of client behaviour.
Critical · IDOR

3. Any SalesRep can delete ANY route assignment

WhereFeatures/DailyRouteAssignments/Controllers/DailyRouteAssignmentsController.cs:121-134 · …/Services/DailyRouteAssignmentService.cs:177-234

The Delete action has no method-level role attribute, so it inherits the class-level list that includes SalesRep. In the service, only the Supervisor branch checks ownership; every other role (including SalesRep) falls through to an unconditional soft-delete. A field rep can delete any rep's route assignment, system-wide.

SFA risk YES — real Operational sabotage / accidental wipe of another rep's day plan.
Critical · IDOR

4. SalesInvoices — any authenticated user reads / enumerates any invoice

WhereFeatures/SalesInvoices/Controllers/SalesInvoicesController.cs:14,32-61

Controller is a bare [Authorize]. GetById fetches purely by primary key with no ownership scoping, and GetList trusts a client-supplied ?distributorId= with no caller-scope validation. A distributor can enumerate another distributor's financial invoices by iterating IDs.

SFA risk YES — real Confidential financial data disclosure across tenants/distributors.

WhereFeatures/GRNs/Controllers/GrnsController.cs:37-65

Same shape as #4: bare [Authorize], GetById by PK with no ownership check, GetList honours a client distributorId with no scope check. (The distributor portal endpoints at :147-161 do enforce ownership correctly — so the fix pattern already exists nearby.)

High findings

High · Duplicates

6. Concurrent idempotency stores throw — a successful write surfaces as HTTP 500

WhereInfrastructure/Caching/PostgresIdempotencyService.cs:27-39

StoreAsync does a blind INSERT on the primary-key column with no upsert / ON CONFLICT. When the #1 race lets two requests complete, the second store violates the PK and throws — and because this runs after the controller succeeded, the user sees a 500 for work that actually committed, confusing retry logic.

SFA risk YES Masks success as failure → encourages yet another retry → compounds #1/#2.
High · Duplicates

7. Idempotency key ignores the request body and path

WhereCommon/Middleware/IdempotencyMiddleware.cs:22-34

The cache key is only {userId}:{rawKey} — no path, method, or body fingerprint. If a client ever reuses a key across two different mutations, the server replays the first response for the second request, silently skipping the second operation and returning a stale body.

SFA risk YES Most dangerous on web, where one helper would generate keys for arbitrary endpoints.
High · Duplicates

8. The web client never sends an idempotency key

Wheresfa_web/lib/api/client.ts:73-93

createIdempotencyKey() is defined but never called anywhere in sfa_web. Every web POST/PUT/DELETE flows through the middleware with no key and gets zero duplicate protection (the middleware short-circuits when the header is absent).

SFA risk YES Admin/portal mutations from the browser are entirely unprotected against double-submit.
High · IDOR

9. Billings GET-by-id & list — no ownership scoping for non-portal callers

WhereFeatures/Billings/Controllers/BillingsController.cs:36-78 · …/Services/BillingService.cs:368-387

Bare [Authorize]; GetById fetches by PK and GetList is a passthrough that trusts client salesRepId/distributorId/outletId. A rep or distributor can read any bill (amounts, outlet, geo, rep names) by id. The write/state transitions (Cancel/Approve/Reject/payment/cash) do check ownership correctly — it's the reads that leak.

SFA risk YES Cross-rep / cross-distributor financial disclosure.
High · IDOR

10. Users GET-by-id — any authenticated user reads any user's profile

WhereFeatures/Users/Controllers/UsersController.cs:30-37

Bare [Authorize] with no self-or-Admin check (unlike UpdateUser, which does enforce it). Discloses PII — name, username, email, phone, role, distributor. Good news: UserDto does not expose password hash, PIN, or tokens, so this is PII + org-structure disclosure, not a credential leak.

SFA risk YES Staff PII exposure / org-chart enumeration.
High · Integrity

11. Soft-deleted rows leak through unfiltered single-record fetches

WhereDistributorRepository.GetByIdAsync, PurchaseOrderRepository (get/list/counts), UserReportingLineRepository, UserGeoAssignmentRepository (incl. stats counts)

These entities have no global query filter, and several methods omit the !IsDeleted predicate that list paths use — so a soft-deleted distributor is still fetchable by id, and deleted/inactive purchase orders leak into get, list, and status counts. Most other entities do have a global HasQueryFilter; these are the inconsistent stragglers.

SFA risk YES Deactivated distributors/orders reappear in user-facing reads and dashboards counts.

Medium · Concurrency

12. Inconsistent optimistic-concurrency coverage (lost updates)

Only the geo hierarchy (Region/Area/Territory/Division) has an xmin rowversion. Outlet, User, Distributor, Product have none — two managers editing the same record = silent last-writer-wins. Two Billing setters (UpdatePaymentTypeAsync, UpdateCashCollectedAsync) take no lock and no token either. Note: status transitions are saved by advisory locks, and all money/stock totals go through FOR UPDATE, so there is no financial corruption risk here — just plain lost edits on master data.

WhereAppDbContext.cs:749-847 · BillingService.cs:731-769
Medium · Duplicates

13. Idempotency is opt-in per request — most mutating endpoints unprotected

The only two clients that actually send the key are mobile billing and not-billing create. Billing status transitions, all PurchaseOrder/SalesInvoice/GRN/Stock/StockTaking mutations, and location pings send no key. Some have their own DB guards; others (PurchaseOrders, stock adjustments) rely on nothing.

Wherebills_remote_datasource.dart:37 · not_billings_remote_datasource.dart:30 (only call sites)

The unique indexes on Distributor Email/Phone are not filtered by IsDeleted, so a soft-deleted distributor's email/phone can never be reused — inconsistent with Users, which deliberately frees them on soft-delete. Behavioral inconsistency, not corruption.

Low findings & hardening notes

Low

19. Bill / GRN / PO numbers develop gaps when the retry strategy re-runs a transaction

On a transient-failure retry, the whole lambda re-runs including nextval, which is non-transactional — so sequence values are burned, producing gaps in bill numbers. Not a collision or oversell risk, but finance teams often expect contiguous numbering. BillingService.cs:275

Low

20. Idempotency "anonymous" fallback shares a key namespace

With no user claim the scope falls back to literal anonymous. Practically limited because all audited mutating endpoints require auth, but any future [AllowAnonymous] POST with a client key could read another anonymous caller's cached response. IdempotencyMiddleware.cs:30-34

Low

21. Cached-response replay drops headers (incl. Location) and forces JSON

A replayed 201 omits the Location header and always sets application/json. Clients read the body, so low impact — but the replay is not a faithful copy. IdempotencyMiddleware.cs:46-49

Low

22. No upper bound on money / quantity values in validators

Validators enforce non-negative but no ceiling; a pathological value (e.g. 10¹⁵) is constrained only by the decimal(18,x) column and could overflow a summed total. Low likelihood.

Low

23. Distributor geographic FKs (Territory/Area/Region) are nullable

A distributor with a null territory won't appear in territory-scoped rep stock-sync queries — operational gap, weakens the "1 distributor per territory" guarantee when unset. Documented as intentional for legacy rows.

Low

24. No global authorization fallback policy; Swagger served in Production

AddAuthorization() has no FallbackPolicy, so a future controller missing [Authorize] would be public by default. app.UseSwagger() runs unconditionally, exposing the full API surface in Production. Program.cs:129,248-249
