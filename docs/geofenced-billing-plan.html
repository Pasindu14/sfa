Proximity-Gated Billing

Only let a rep bill an outlet when they have physically reached it — outlets outside a server-configurable radius of the rep's live GPS position are hidden from the order screen and rejected by the server. Launches at 1 km, then tightened gradually from the backend — no app release per change.

1 What you asked for

In the rep's mobile app, the Create Bill screen lists outlets to bill. Today it shows every outlet on the rep's assigned route. You want it to show only the outlets the rep is standing near — within a given range of their current coordinates.

Rep at 6.9271, 79.8612 → Radius = 1 km
server-set · tightened over time → Show only outlets whose distance ≤ 1 km → Far outlets: hidden & un-billable

Your example "78.215, 78.455 → 1 km shops only" is exactly this: compute the straight-line distance from the rep to each outlet, keep the ones within the limit.

Agreed approach (locked): launch with the radius at 1 km, keep it configurable from the server, and decrease it step by step over time as confidence grows (e.g. 1 km → 500 m → 300 m). Because the value lives on the backend, each tightening is a config change — no new app build, no store release. This makes the server-driven radius (Phase 4) a day-one requirement, not a later nice-to-have.

3 How the distance is calculated

The math (no new dependency)

The geolocator package — already a dependency — ships a Haversine helper that returns the great-circle distance in metres between two coordinates:

final meters = Geolocator.distanceBetween(
  repLat, repLng,        // from CreateBillState
  outlet.latitude,
  outlet.longitude,
);
final isNearby = meters <= radiusMeters; // 1000.0

No backend call, no map SDK, works fully offline — pure arithmetic on two points.

Why straight-line is fine here

  • A 1 km gate is generous relative to GPS error (~5–30 m outdoors), so road-vs-crow-fly difference is irrelevant.
  • It's instant and offline — reps often bill with poor connectivity.
  • We sort the kept outlets nearest-first and print the distance on each card, so the rep sees "Kumar Stores · 120 m" and picks confidently.

As you tighten the radius: straight-line stays correct, but the safety margin shrinks. Below a few hundred metres the limiting factor becomes outlet-coordinate accuracy (captured once at outlet creation) plus live GPS error — which is exactly why Phase 2 (accuracy gating) and the distance audit matter more the smaller the radius gets.

Phase 2 — Trust the GPS (accuracy & anti-spoofing)

~½ day

A 1 km gate is only meaningful if the fix is real. Field reps have a strong incentive to fake location, so we harden the capture step.

ConcernHandling
Poor accuracy (e.g. ±400 m indoors) Reject fixes where position.accuracy > accuracyThresholdMeters (e.g. 100 m); add LocationCheckStatus.lowAccuracy + a "Improve GPS signal / Retry" view.
Mock-location apps (fake GPS) position.isMocked → block with a distinct message, and flag the attempt for the server (see Phase 3). Optional but recommended for a sales-rep app.

Phase 3 — Server-side enforcement (un-bypassable)

~½ day
Why this matters: the client filter is UX, not security. A modified APK could bill any outlet. The server already receives the rep's coords and the outlet's coords — so it can and should verify the distance before accepting the bill.

The check drops cleanly into BillingService.CreateAsync, right after the outlet is loaded (line 52) and before coords are stamped (line 246):

// ① Validate outlet  (existing)
var outlet = await _billingRepository.GetOutletAsync(request.OutletId, ct)
    ?? throw new NotFoundException("Outlet", request.OutletId);

// ②  NEW — proximity guard
if (_geoOptions.EnforceProximity && request.Latitude is { } lat && request.Longitude is { } lng)
{
    var meters = GeoMath.HaversineMeters(lat, lng, outlet.Latitude, outlet.Longitude);
    if (meters > _geoOptions.RadiusMeters + _geoOptions.ToleranceMeters)
        throw new ValidationException(
            "OUTLET_OUT_OF_RANGE",
            $"You must be within {_geoOptions.RadiusMeters} m of the outlet to bill it.");
}

Phase 4 — Server-driven radius · required for your tighten-over-time plan

~½ day
Not optional in your case. Because you start at 1 km and decrease gradually, the radius must live on the server — otherwise every tightening step is a new app build pushed to every rep's phone. Ship this together with Phases 1 & 3, not after.
  • One server setting — BillingGeoOptions.RadiusMeters — is the source of truth for the server range check.
  • Expose that same value to the app via geofenceRadiusMeters in the daily outlet/route sync payload the app already pulls. The app uses it for the client filter, falling back to the bundled 1000.0 constant only when offline before the first sync.
  • One knob drives both client filter and server check → they can never drift apart. Lower it in the backend, and reps pick up the tighter radius on their next daily sync.
  • Recommended (the safety net for tightening): store DistanceFromOutletMeters on each Billing row. This is what lets you step down safely — you can see the real spread of billing distances and only drop to 500 m once you confirm, say, 98% of bills are already under 400 m.
  • Optional: per-outlet override GeofenceRadiusMeters for malls / large venues that legitimately need a wider radius.

8 Decisions I need from you

1. Radius — DECIDED. Launch at 1 km, one global value for all outlets, server-configurable, then decreased gradually over time. Per-outlet overrides deferred until a real need appears.
2. Hard block vs. warn-first. On the server, reject out-of-range bills immediately, or run a 1-week "warn & log" period to measure false-positives before enforcing?
