Data Consistency & Accuracy Review · SFA API

Beyond Security: Will the Numbers Always Be Right?

The security pass is done. This is the companion review — the things that don't leak data but quietly make it wrong: stale prices, duplicate bills, un-reconciled stock, and day-boundary drift. Focus: consistency, accuracy, no duplicates. Not security.

Scope: D:\Github\sfa\sfa_api (billing, stock, sync, reporting) Stack: .NET 8 · EF Core · PostgreSQL · Flutter Method: 2 parallel full-source surveys, code-cited Date: 2026-06-24

The honest bottom line

What is already accurate: The arithmetic core is sound. Money is decimal(18,2) with explicit Math.Round(x, 2) everywhere, totals sum with no float drift, dates store as UTC, hot queries are properly indexed, list reads avoid N+1, and bill cancellation correctly reverses stock. Stock-taking captures variance under a row lock. These are the parts that usually go wrong in SFA apps — and yours are right.

What can still make data wrong: Billing trusts the price the phone sends — it never re-checks the product master, so a stale offline price list writes wrong revenue with no error. Worse, there are two independent pricing systems (flat dealer price for reps, category-tier price for distributors) that nothing reconciles. There is no soft duplicate-bill detection (a reinstall or web re-submit can create a true duplicate). Credit limits are stored but never enforced. There is no automated stock reconciliation (GRN in vs billed out vs on-hand). None of these throw an exception — that's exactly why they're dangerous: the books look fine until month-end doesn't tie out.

High impact — these silently corrupt the books

High · Accuracy

1. Billing trusts the price the phone sends — it never re-checks the product master

WhereFeatures/Billings/Services/BillingService.cs:190–234

The line-item builder takes item.UnitPrice straight from the request and stores it. The product's own DealerPackPrice / DealerCasePrice / Mrp are never read during billing.

UnitPrice  = item.UnitPrice,                                    // client value, stored as-is
totalPrice = Math.Round(item.Quantity * item.UnitPrice, 2);     // no comparison to Product price

The math is perfect — but it's perfect math on an unverified input. The mobile app caches a price list and can bill offline. If that cache is a day (or a price-revision) stale, every bill writes the wrong revenue, and the system reports it as fact. This is the classic distribution-software failure: the totals all add up, but they add up to the wrong number, and you only discover it when the distributor's payment doesn't match your invoice.

Impact Wrong revenue, silently Affects every bill, not an edge case. No alarm.
Fix (decide the policy first): If reps cannot negotiate price → server re-fetches the current Product price and overrides the client value entirely; the phone's price becomes display-only. If reps can negotiate → keep the entered price but also snapshot the master price onto the BillingItem (e.g. CatalogUnitPrice) so you can report the variance and detect stale-list drift. Either way, the server — not the phone — must be the price source of truth.
Business question to answer: For Uswatte, is the dealer price fixed per product, or can a rep give a special price at the shop? That single answer decides "override" vs "snapshot & flag."
High · Duplicates

2. A reinstall or web re-submit can create a genuine duplicate bill

WhereAppDbContext.cs:775 (only ClientBillId is unique)

Billing's only duplicate backstop is the filtered unique index on ClientBillId (the phone-generated UUID). That correctly stops double-taps and retry-after-timeout from the same install. It does not stop a duplicate when the key changes:

e.HasIndex(x => x.ClientBillId).IsUnique().HasFilter("\"ClientBillId\" IS NOT NULL");

Scenario: rep submits a bill → app is reinstalled / cache cleared / the same bill is re-entered from the web → a new ClientBillId is generated → the unique index does not fire → two bills for the same sale.

Why a hard constraint is the WRONG fix Nuance You cannot add a unique (SalesRepId, OutletId, BillingDate) index here — a rep legitimately bills the same shop twice a day (a morning order and an evening top-up are both real). NotBilling can have that hard index (AppDbContext.cs:920) precisely because a shop is "not billed" at most once a day. Billing is different.
Fix: Soft duplicate detection, not a DB constraint. On create, look for an existing bill with the same rep + outlet + a near-identical total within the last N minutes; if found, return it (or require an explicit "yes, this is a second order" confirmation flag). This closes the reinstall/web gap without blocking legitimate repeat visits.

Deep dive — the pricing architecture

The single price-trust finding above is the tip of a larger issue: pricing is split across two unrelated engines, and which one applies depends on which channel the order comes through. This is where "consistent & accurate" is most at risk.

High · Accuracy / Fraud

12. Return-line prices are freely editable on the phone, with no ceiling

Wherecart_row.dart:392-457 (edit dialog) · create_bill_bloc.dart:189 (clamp)

Sale lines are locked to the catalog price — good. But for return lines the rep can type any unit price in a dialog, clamped only to >= 0:

return l.copyWith(unitPrice: e.unitPrice.clamp(0.0, double.infinity));  // no upper bound

Nothing ties a return price to the original sale price, the catalog price, or MRP. A rep can credit a return at Rs.10,000 for goods that sell at Rs.100 — directly reducing the bill total / issuing outsized credit. The server's only guard is the blanket <= 1,000,000 ceiling added in the security pass.

Impact Most exploitable hole found No policy decision needed to cap it — this is a clear correctness fix.
Fix: Cap a return line server-side at the product's current catalog price (or, ideally, the price it was actually sold at). Validate in the billing service, not just the UI.
Medium · Integrity

13. MRP enforces nothing, and prices have no cross-field validation

Mrp (Maximum Retail Price — a legal ceiling in FMCG) is captured, synced to mobile, shown in the web grid… and used by zero code paths. Billing never checks the billed price against it; neither pricing engine does. Separately, the three price fields are each validated only as 0 ≤ x ≤ 1,000,000 — independently. Nothing asserts the natural invariants:

DealerPackPrice  ≤  Mrp
DealerCasePrice  ≈  DealerPackPrice × PiecesPerPack

So Mrp = 10, DealerCasePrice = 500 saves cleanly — nonsensical (you'd charge the dealer 50× the consumer's legal ceiling), and a fat-fingered entry passes straight through.

WhereCreateProductValidator.cs:29-39 · Product.cs:17-19
Watch the default: Mrp = 0 passes GreaterThanOrEqualTo(0). Any future "billed price ≤ MRP" rule would reject all billing for products still at the 0 default — so enforcement and making MRP required/non-zero are the same decision.
Fix: Add cross-field validators (Mrp >= DealerPackPrice, case≈pack×pieces) — a pure win regardless of policy. Then decide MRP's billing role: hard cap, soft warn, or explicitly display-only.
Low

14. Pack-from-case price is derived with floating-point division — cents leak

When a rep sells by the case, mobile derives the per-pack price as a Dart double: quantity_dialog.dart:238 _casePrice / _packsPerCase. Case Rs.100 ÷ 12 packs = 8.333… → stored to decimal(18,2) = 8.33 → 12 × 8.33 = 99.96 ≠ 100. Every case sold as derived packs leaks a few cents that never reconcile back to the case price. Fix: compute in decimal, or send the case price + pack count and let the server do exact arithmetic.

Low

15. Latent precision boundary: category price (18,4) vs billing price (18,2)

Category prices are decimal(18,4); BillingItem.UnitPrice and the dealer fields are decimal(18,2). The two engines don't connect today, so this is dormant — but the moment anyone unifies pricing (finding #11), there's a silent rounding step at the boundary. Fix: settle on one precision for "a price" across both tables before merging them. ProductCategoryPrice.cs:12-13 · AppDbContext.cs:439

Time & timezone

The server runs in UTC; the business runs in Sri Lanka (UTC+5:30). Because Sri Lanka is ahead of UTC, the server clock is always behind the rep's — so the danger window is the early morning, 00:00–05:30 SL, where the server's UTC date still reads yesterday. Outside that window the two dates agree.

✅ Remediated (2026-06-24) — all time findings fixed or accepted

Root cause was scattered use of DateTime.UtcNow for business dates. Fixed by routing every business "today/this year" through one server helper, SriLankaTime (Common/Extensions/SriLankaTime.cs); localizing the two mobile timestamp displays with .toLocal(); and pinning the web to Asia/Colombo via one helper (lib/utils/datetime.ts) across ~34 files. API suite green: 873 unit + 500 integration (1 known skip). Audit stamps, token expiry and cache TTLs were deliberately left on UTC — those are absolute instants, not business days.

OK

What was already correct (and deliberately left alone)

Mobile sends the rep's local business date for bills/not-billings — the server fallback rarely even fires.
Offline sync preserves the original creation date when a bill syncs the next morning.
Location pings use .toUtc() — correct for absolute event instants.
Report date-range filters compare DateOnly to DateOnly consistently.
All audit stamps (CreatedAt/UpdatedAt), token expiry, cache TTLs stay UTC — absolute instants, not business days.
Date-only mobile displays were not localized — applying .toLocal() to a midnight date-only value would have shifted the day.
The fix in one line: a business date now has exactly one source of truth per tier — SriLankaTime on the API, .toLocal() at the mobile display edge, and Asia/Colombo via datetime.ts on the web. None of them can disagree with each other anymore.

Worth a deliberate decision

Each of these is a business-policy call as much as a code change — but leaving them undecided is itself a (risky) decision.

Medium · Business rule

3. Credit limit is stored but never enforced

Outlet.CreditLimit exists (Outlet.cs:22) but a full-text search shows it is never read in the billing or order path — a rep can bill any amount to a shop that is already over its limit. For a distribution business running on credit, this is the difference between controlled and uncontrolled receivables.

Fix: Decide the policy — block over limit, warn-and-allow, or allow-but-flag for supervisor. Then compute the outlet's outstanding balance (sum of unpaid bills) and compare to CreditLimit at create time. Even "warn-and-flag" is a large improvement over silence.
Medium · Accuracy

4. No automated stock reconciliation (GRN in − billed out vs on-hand)

You have StockTaking (physical count vs system) which is great for catching shrinkage — but there is no system-internal reconciliation that proves the ledger is self-consistent: opening + GRN received − billed − adjustments + returns = current QuantityOnHand. If a bug ever mis-posts a stock transaction, nothing detects the drift until a human counts the warehouse.

Fix: A periodic (nightly or on-demand) reconciliation report per distributor+product that recomputes expected on-hand from the transaction ledger and flags any row where expected ≠ actual. It's read-only and cheap, and it turns silent data bugs into a dashboard line. (You already have the BackgroundService pattern from the cleanup jobs to host it.)
Medium · Accuracy

5. "Today's sales" buckets by UTC date, not Sri Lanka business day

Storage as UTC is correct. The risk is bucketing: when BillingDate falls back to DateOnly.FromDateTime(DateTime.UtcNow) (BillingService.cs:255), a bill entered between midnight and 05:30 Sri Lanka time lands on the previous UTC day — because SL is UTC+5:30. Daily route totals and rep leaderboards would then attribute it to the wrong business day.

Status Probably fine, verify Mitigated if the mobile client always sends the local business date and never relies on the server default. Worth confirming, not assuming.
Fix: Make the business day explicit — either require the client to send the local BillingDate (and validate it), or convert UtcNow to Asia/Colombo server-side before taking the date. Pick one and apply it everywhere reports bucket "by day."
Medium · Business rule

6. A rep can give a 100% discount (free goods) with no approval tier

Discounts are correctly bounded to 0–100%, but 100% is allowed at both item and bill level — a rep can zero out a bill. There is no per-role ceiling and no approval threshold. Whether that's acceptable is a policy question, not a bug.

Fix (if desired): Cap discount by role (e.g. rep ≤ 5%, supervisor ≤ 15%), or require approval above a threshold. Enforce in the validator/service, not just the UI.
Medium · Accuracy

7. Duplicate customer (outlet) master records are possible

From the prior audit (#17, accepted): Outlet.NicNo has only a non-unique index (AppDbContext.cs:367). Two reps registering the same shop create two outlet records — which then split that shop's sales history in two, quietly making per-customer analytics inaccurate. This is an accuracy concern even though it was acceptable from a security view.

Fix: A soft dedup check at outlet creation (same NIC / phone / very-close coordinates → surface "possible existing outlet" before saving). A hard unique index may be too strict if one owner runs several outlets — so make it a guided warning, like finding #2.

Hardening & housekeeping

Fixed

8. RefreshTokens table grows forever

Idempotency keys and audit logs are purged on a schedule, but RefreshTokens has an ExpiresAt and no cleanup job — expired/consumed/revoked rows accumulate indefinitely. Not a correctness bug, but unbounded growth on an auth table. Fix: add a daily cleanup deleting tokens past expiry + a grace window, reusing the existing BackgroundService pattern.

✅ Fixed (2026-06-24): Features/Auth/Services/RefreshTokenCleanupService.cs — daily sweep deletes tokens where ExpiresAt < UtcNow − RefreshTokenRetentionDays (default 7d grace); a single predicate covers expired/consumed/revoked since all carry ExpiresAt. Registered in Program.cs alongside the other cleanup hosted services.
Fixed

9. Concurrent edits to master data are last-writer-wins

From the prior audit (#12, accepted): Outlet, User, Distributor, Product have no xmin rowversion (only the geo hierarchy does). Two managers editing the same outlet simultaneously = one silently overwrites the other. No financial impact (money/stock go through locks), but master-data edits can be lost. Fix (if it bites): add rowversion to those four, mirroring the Region/Territory pattern already in the codebase.

✅ Fixed (2026-06-24): full Region/Territory pattern replicated on all four entities — uint RowVersion mapped to xmin in AppDbContext.cs, exposed on each read DTO, required (non-zero) on the update request, applied as the EF concurrency OriginalValue in the service, and surfaced as 409 CONCURRENCY_CONFLICT on a stale write. The web edit forms (user, outlet, distributor, product) round-trip a hidden rowVersion, matching the existing area form. Migration 20260624062113_AddRowVersionToMasterData is no-op DDL (the script is only the history insert), so it adds nothing to the blocked Routes migration risk. Mobile: no change — it never issues PUT/PATCH for these entities. SQLite test provider patched in TestAppDbContext.cs (xmin → INTEGER, concurrency disabled), so the token path runs only against real PostgreSQL.
Fixed

10. Reports don't state whether inactive entities are included

Deactivating a parent (distributor/product) does not cascade — children stay in the DB and the system relies on per-query IsActive filters. That's a reasonable design, but it means a report's number depends on whether each query remembered to filter. Fix: be explicit in reporting queries about active-vs-all, and document it, so "total products" and "total sales" can't disagree about what counts.

✅ Fixed (2026-06-24): the policy is now written down in .claude/docs/reporting-conventions.md — two distinct universes (current master-data counts vs historical financial facts) that legitimately differ, with the rule that a money aggregate must filter the bill's own state and never the referenced entity's current IsActive. The two key billing aggregation sites (BillingRepository.GetRepMonthlySalesTotalAsync cluster and GetOutletSummaryRawAsync) now carry explicit comments stating their stance, and the convention is linked from sfa_api/CLAUDE.md.
