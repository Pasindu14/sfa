1. What a bin card requires vs. what exists

A bin card (stock ledger card) tracks, for one product at one location, every movement with a running balance. Here is each required property mapped to the codebase.

2. The ledger model (the bin card "row")

sfa_api/.../Features/Stock/Entities/StockTransaction.cs

Each row is one line on the bin card. The key fields:

  • TransactionType + Direction — what happened and whether it added or removed stock
  • Quantity (always positive), QuantityBefore, QuantityAfter — the running balance, snapshotted at post time
  • StockType — Normal saleable stock vs. company Free-Issue (FOC) pool
  • ReferenceType / ReferenceId — drill-back to the GRN, Bill, or Stock-Take that caused the move
  • TransactedBy / TransactedAt / Notes — full audit trail
Why this is the right design: because QuantityAfter is persisted on every row, you can render a bin card for any date range without recomputing from the beginning of time — the closing balance of the last row in the range is authoritative, and the opening balance is the QuantityBefore of the first row in the range.

Verified: Billing, free issues (both funding sources), resaleable returns, bill cancellations, bill rejections, GRN receipts, and stock-take adjustments all post to the same ledger. There is no code path that changes a distributor's stock without writing a ledger row — the balance and the ledger cannot drift apart.

4. How a movement is posted (atomic & lock-safe)

sfa_api/.../Features/Billings/Services/BillingService.cs — CreateAsync (sale line)
// 1. pre-check availability (fast fail, collects ALL shortages) BeginTransaction() GetStockForUpdateAsync(distributorId, productId, Normal) // SELECT … FOR UPDATE — row lock DeductStockAsync(qty, Sale, ref="Billing", billId) ├─ QuantityBefore = stock.QuantityOnHand ├─ guard: QuantityAfter < 0 → INSUFFICIENT_STOCK ├─ stock.QuantityOnHand = QuantityAfter // update running total └─ INSERT StockTransaction { … Direction=Out … } // append ledger row Commit()

The same pattern wraps GRN confirmation and stock-take adjustments. Posting is also guarded by a PostgreSQL advisory lock (e.g. billing:create:{repId}, grn:confirm:{grnId}) so concurrent operations serialize cleanly rather than racing on the same balance.

6. Gaps to close for a complete "Bin Card" feature

1  Bin Card screen + printable report (UI)

The data and API exist; build a screen that, given a distributor + product (+ optional date range), shows: Opening balance → each movement (date, type, ref, in, out, balance) → Closing balance. The web stock feature already has the action wired — it needs a drill-in (e.g. click a product row → ledger drawer/page) and an export-to-PDF/Excel.

2  Period opening/closing endpoint

Today the transactions endpoint is page-based and sorted newest-first, with no date filter. A true bin card wants ?from=&to= plus computed opening balance (the QuantityBefore of the first row in range) and closing balance. This is a small repository addition — the data is already there in QuantityBefore/QuantityAfter.

3  Wire the Opening transaction type

Opening / go-live balances should be posted as Opening (Direction=In) ledger rows so the card has a proper starting point instead of stock appearing only from the first GRN.

4  Wire the Damage transaction type

Physical damage or expiry discovered at the distributor (not via a sales-return line) currently has no way to leave the ledger. A "stock write-off" action posting Damage (Direction=Out) would close this. Note: damaged/expired sales returns are intentionally recorded on the bill without a stock move — confirm that matches the desired physical-stock policy.

Data-integrity watch-item: because Damage/Expire returns deliberately skip stock movement, physically unsaleable returned goods are recorded financially (on the bill) but never decrement a stock pool. If you expect a "damaged goods" balance to track at the distributor, that needs an explicit policy decision and a wired movement type.

7. Summary

  • Tracking everything happening to distributor stock?  YES — receipts, sales, both kinds of free issue, resaleable returns, bill cancel/reject reversals, and stock-take corrections all post to one immutable, concurrency-safe ledger with full audit + source linkage.
  • Is the ledger query-able as a bin card?  YES (API) — a per-distributor, per-product transaction history endpoint already returns the card data.
  • Is there a Bin Card feature (UI + period report)?  NOT YET — needs a screen/report, a date-range opening/closing endpoint, and the Opening + Damage movement types wired in.

So the feature you described is ~80% already in place at the data layer. The remaining work is primarily presentation and two small posting flows — not a from-scratch inventory engine.
