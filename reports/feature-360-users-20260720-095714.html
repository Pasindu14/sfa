What it does

API — Admin-gated CRUD over User (Id/Name/Username/Email/Phone/Role/DistributorId/DeviceId/IsActive + xmin). Self-or-Admin read/update, owner-only change-password, Admin-only reset-password/activate/deactivate/delete (soft). Username & Email lowercased on write; filtered unique indexes on Email/Username/Phone. Distributor role must carry a valid DistributorId; SalesRep must carry a DeviceId.

Web — TanStack Query + Zustand reference feature. Server-side paginated table with search + role filter, create/edit dialog, reset-password dialog, activate/deactivate/delete confirmations. Zod mirrors the API validators; edit form round-trips rowVersion.

Mobile — no management UI. Decodes the role claim from the JWT (safe fallback to salesRep) and pushes the FCM device token to /users/me/fcm-token.

Data Integrity — the business runs on this

CRITICAL  The web edit form silently destroys / cannot preserve DeviceId.

The API read DTO UserDto (DTOs/UserDto.cs:3-16) does not include DeviceId, yet the web UserDto type declares deviceId?: string (schema.ts:97) and the edit form binds it (user-form.tsx:202-214). Because the field is never returned, the edit form always loads it as empty and submits deviceId: "", and the API update path then writes that empty value straight onto the entity (UserService.cs:123 — user.DeviceId = request.DeviceId). Two concrete failures:

  • Non-rep edit → silent data loss: every edit of any Admin/NSM/RSM/ASM/Supervisor/Distributor user overwrites their stored DeviceId with empty — the real value is destroyed without warning.
  • SalesRep edit → hard block / corruption: UpdateUserValidator.cs:36-38 requires DeviceId for SalesReps, so editing any of the ~500 field reps returns 400 unless the admin types a device ID they can never see — and any value they invent overwrites the rep's genuine device binding. The web updateUserSchema (schema.ts:54-58) has no matching rep-required rule, so the failure only surfaces server-side.

Fix: add DeviceId to UserDto + MapToDto so the form can round-trip it; mirror the SalesRep-required rule in updateUserSchema. (Contract fix — API + web in one pass.)

MEDIUM  Case-sensitive duplicate pre-check vs. lowercased storage. UserService.cs:40,43,90,93 checks ExistsByUsername/EmailAsync(request.Username/Email) with the raw mixed-case input, but persists .ToLowerInvariant() (lines 64-65, 119-120). A mixed-case duplicate slips past the friendly check and instead trips the DB unique index → generic DUPLICATE_RESOURCE 409 with fields:null rather than the field-specific "Username already exists". No duplicate row is created — the filtered unique index (AppDbContext.cs:115-117) protects data — so this is error-quality degradation, not corruption.

Verified safe: soft-delete correct (UserRepository.cs:91-100 sets IsDeleted=true+IsActive=false, no context.Remove()); global query filter !IsDeleted (AppDbContext.cs:124); filtered unique indexes free email/username/phone on delete; optimistic concurrency round-tripped on the update path (ApplyConcurrencyToken, UserService.cs:116 + updateUserSchema.rowVersion); no double-submit (submit button disabled on isPending); mobile userRoleFromString has a safe default (no fromJson crash on unknown role).

Cross-Impact Analysis — if the contract changes, what breaks

  • Adding DeviceId to UserDto (the fix): non-breaking — web form gains the value it needs to preserve; mobile unaffected. Update MapToDto (UserService.cs:215) in the same change.
  • Renaming/retyping the role enum values: breaks web roleEnum (schema.ts:3), the table role selector (user-table.tsx:88-95), the columns label map, and mobile userRoleFromString (user_role.dart:3). Mobile fails silently — an unknown value defaults to salesRep, a privilege downgrade rather than an error.
  • Retyping rowVersion: breaks updateUserSchema.rowVersion (schema.ts:56) and the hidden form field (user-form.tsx:233-251) → optimistic concurrency protection lost on edits.
  • Entity/migration: User carries no denormalized ancestor IDs, so geo re-parent cascades do not touch it. The filtered unique indexes on Email/Username/Phone (AppDbContext.cs:115-117) are the only integrity guard behind the app-level duplicate checks — do not drop them.
  • One-sided validation drift: the SalesRep-DeviceId rule and the Distributor-DistributorId rule live in the API validator; only the Distributor rule is mirrored in Zod. The DeviceId rule is not — that gap is the root of Finding #1.
