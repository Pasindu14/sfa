01Bottom line

Is this API built to an industry-standard pattern? Yes — clearly. Is every feature production-correct today? No — a set of authorization, soft-delete, and concurrency gaps need closing first.

8/10 Engineering
Standard

The architecture, error handling, observability, persistence and testing are genuinely strong — above the typical line-of-business bar.

What holds it back from "production-ready" is not the design. It's a recurring set of issues in the hand-extended read paths and the newest transactional code: broken object-level authorization (IDOR), soft-delete leakage, unbounded pagination, and two places where stock-mutation concurrency is not actually safe. None require a redesign — they're targeted fixes on solid foundations.

02What is genuinely industry-standard

The majority of the codebase is strong. These are not box-ticks — they were verified in code across many features.

Architecture & consistency

Clean vertical slices (Features/{X}/) with a disciplined Controller → Service → Repository split, applied uniformly across ~26 features. The brand-new Bin Card feature is structurally indistinguishable from the year-old Users feature.

Signal: uniformity is what separates a standard codebase from one that merely contains standard snippets.

Error handling

A typed SFAException hierarchy maps centrally to 400/401/403/404/409/422/429/503 with a 500 INTERNAL_ERROR fallback that never leaks stack traces. Model-validation and thrown errors share one ApiResponse/ApiError envelope.

Observability

OpenTelemetry traces + metrics, Serilog → Seq with correlation IDs, a dedicated slow-query interceptor (≥500 ms), audit interceptor that redacts secrets. Real, not cosmetic.

Where: Program.cs:114-124, SerilogConfig.cs, AuditInterceptor.cs

Persistence

Pooled DbContext with tuned Npgsql + retry, xmin optimistic concurrency on Areas, ~82 purposeful migrations and indexes, money stored as decimal(18,4). Billings & GRN use real SELECT … FOR UPDATE row locking inside the transaction.

Where: Program.cs:64-76, StockRepository.cs:110-123

Auth design

Refresh tokens stored hashed (SHA-256), CSPRNG-generated, rotated on use with family-reuse detection and LogCritical alerting. Login is BCrypt + generic error (no user enumeration via message) + rate limited.

Testing discipline

~1,250 tests (xUnit + Moq + FluentAssertions + WebApplicationFactory), correctly shaped as a pyramid. Even the one skipped integration test is honestly skipped with a documented reason, not faked green.

03Systemic themes — the highest-value findings

These patterns recurred across independent auditors who never saw each other's work — strong signal they are template-level gaps, not one-off slips. Fixing them once, in the shared pattern, fixes them everywhere.

1 · Broken object-level authorization (IDOR)

Many endpoints accept an id or userId from the client and never verify the caller owns it. Write paths are well-scoped from the JWT; read paths frequently are not.

Hit in: Supervisor (3 endpoints, Crit), DailyRouteAssignments GET /{id} (Crit), Outlets, PurchaseOrders, NotBillings, Stock, SalesInvoices.

2 · Soft-delete leakage

The User entity has no global query filter at all, and many repos call IgnoreQueryFilters() on GetById, so "deleted" rows stay readable, editable and counted. Plain (non-filtered) unique indexes also permanently burn an email / code after soft-delete.

Hit in: Users (Crit), SalesTargets, Outlets, Products, ProductCategoryPricings, geo features.

3 · Unbounded pagination

The clamp Math.Clamp(take,1,200) lives in some repositories but not all. Several endpoints pass client pageSize straight to Skip/Take, allowing a full-table pull in one request. page=0 also yields a negative Skip in places.

Hit in: Notifications, MobileSync, SalesInvoices, GRN, Supervisor, Stock GetTransactions; page unguarded in Distributors / Products / PO.

4 · Stock concurrency rests on a fragile lock

Billings & GRN do it correctly (in-transaction FOR UPDATE). But StockTaking's lock does not actually serialize against concurrent billing, and the advisory-lock fallback keys on string.GetHashCode() — process-unstable and collision-prone.

Hit in: StockTaking (Crit), Billings TOCTOU on status transitions, PostgresAdvisoryLockService.cs:14 (all callers).

5 · TOCTOU uniqueness without a DB backstop

Duplicate-name/code checks are read-then-write at the app layer; several are case-sensitive (== instead of ILike) and lack a unique index, so a race surfaces as a raw DbUpdateException → 500 instead of a clean 409. Areas does this correctly — copy that pattern.

Hit in: Regions, Territories, Divisions, Routes, Fleets, Users.

6 · Idempotency is opt-in, not enforced

The IdempotencyMiddleware only acts when the client sends X-Idempotency-Key. Money/stock-critical writes (Billings, GRN confirm, PO create, StockTaking adjust, imports) have no DB-level duplicate backstop, so a double-submit without the header can double-post.

Hit in: Billings, GRNs, PurchaseOrders, StockTaking, SalesInvoices, SalesTargets imports.

Why these matter — in plain terms

The cards above are the summary. Each panel below explains the theme without jargon: what it actually is, the concrete thing that goes wrong, and the simple fix. Click any to collapse.

1Broken object-level authorization (IDOR)Critical▶
In plain terms

Being logged in is not the same as being allowed to see a specific record. These endpoints take an id straight from the client — /orders/55 or ?userId=12 — and return that data without checking the caller actually owns or manages it. (IDOR = Insecure Direct Object Reference.)

What goes wrong

A sales rep — already authenticated — changes /daily-route-assignments/57 to 58 and sees another rep's day. A supervisor calls rep-monthly-sales?userId=<anyone> and reads any rep's sales figures. The login check passed; the ownership check was never made.

The fix

After loading a record, confirm it belongs to (or is managed by) the caller, using the identity from the JWT — never trust a client-supplied id to decide what they may see. Put the check in a shared base-controller helper so a new endpoint can't silently skip it.

2Soft-delete leakageCritical▶
In plain terms

The rule is "never really delete a row — flag IsDeleted=true and hide it." A global query filter is supposed to hide flagged rows from every query automatically. But the User entity has no such filter, and many repositories call IgnoreQueryFilters(), which switches the hiding off.

What goes wrong

You "delete" a user — and they still appear in lists, can still be fetched and edited, and are still counted in stats: a ghost record. Worse, the email's unique index isn't filtered either, so that email is taken forever — if the person rejoins, you can't recreate them. A "deleted" user left IsActive=true can also still pass checks elsewhere.

The fix

Add the IsActive && !IsDeleted global filter to User; audit every IgnoreQueryFilters(); make DELETE also set IsActive=false; and convert the email/username/code unique indexes to filtered (WHERE IsDeleted = false).

3Unbounded paginationHigh▶
In plain terms

A list request sends two numbers — page and pageSize — which the code turns into Skip((page-1)*pageSize).Take(pageSize). Some endpoints trust those numbers blindly instead of capping them.

What goes wrong

pageSize=1000000 turns "give me 25 rows" into "load the entire table into memory" — a few such requests at once can exhaust server memory (an accidental denial-of-service), and it doesn't even need to be malicious, just a buggy client. And page=0 makes the math Skip(-25), an invalid query → HTTP 500.

The fix

Clamp both values before use (pageSize 1–200, page ≥ 1). You already do exactly this (Math.Clamp(take,1,200)) in some repositories — make it one shared paging helper so it's impossible to forget.

4Stock concurrency rests on a fragile lockCritical▶
In plain terms

To deduct stock safely you must do read → subtract → write as one indivisible step, and stop two sales doing it at the same time. The correct tool is a database row lock — SELECT … FOR UPDATE held inside the transaction, which makes everyone else wait until you commit. Billings and GRN do this correctly.

What goes wrong

StockTaking acquires its lock on a different connection than the actual update, so the lock doesn't bracket the work — a concurrent bill can slip in between the count and the adjustment and corrupt the balance. Separately, the fallback lock turns its name into a number with string.GetHashCode(), which (a) changes every app restart, so two server instances don't actually block each other, and (b) can collide, making unrelated operations needlessly wait.

The fix

Hold FOR UPDATE on the same connection/transaction as the work (treat advisory/Redis locks as an optimization, never the integrity boundary), and replace GetHashCode() with a deterministic hash so lock IDs are stable everywhere.

5TOCTOU uniqueness without a DB backstopMedium▶
In plain terms

TOCTOU = "time of check to time of use" — a gap between checking and acting. To enforce unique names the code asks "does 'North' already exist?" then inserts. Between those two steps, another request can insert the same name.

What goes wrong

Two concurrent creates both get "doesn't exist" and both insert → two 'North' regions. Some checks are case-sensitive (== instead of ILike), so "North" and "north" both slip through. And where a unique index does exist, the race throws a raw DbUpdateException → an ugly HTTP 500 instead of a clean 409 "name already taken".

The fix

Copy the Areas pattern — case-insensitive ILike check plus a try/catch that maps a unique-violation to 409 — and ensure every such field has a real database unique index as the unconditional backstop.

6Idempotency is opt-in, not enforcedMedium▶
In plain terms

"Idempotent" means doing the same operation twice has the same effect as doing it once. Creating a bill is not idempotent — call it twice, get two bills. The X-Idempotency-Key header lets the server recognize a retry and return the original result — but it only works if the client sends it.

What goes wrong

A rep on a flaky connection submits a bill; the server succeeds but the response is lost on the way back; the app shows "failed" and the rep taps again → the bill posts twice, stock deducted twice. With no header and no database backstop, nothing stops it. This is extremely common with mobile/offline clients.

The fix

Require the idempotency key on money/stock-mutating endpoints (reject if missing), and/or add a natural database unique constraint — e.g. on BillingNumber, or (rep, outlet, date) — so a duplicate physically can't be saved even when the header is absent.

The pattern behind the pattern

Notice that the gaps cluster in hand-written read endpoints and the newest transactional features — not in the generator-produced CRUD core. The framework is right; new code added outside the scaffold path doesn't automatically inherit every guardrail (auth scoping, pagination clamp, query-filter consistency). The durable fix is to bake these into the feature template and a base controller, not to chase each instance.

05Recommended fix order

Sequenced by blast radius and effort. The first item is non-negotiable and quick; items 2–4 are the systemic fixes that retire many findings at once.

  1. Rotate & purge committed secrets.DB password, JWT key, Upstash token in appsettings*.json + the admin JWT in sfa_api/CLAUDE.md. Rotate all, move to env/secret manager, scrub history. Rotating the JWT key also invalidates the leaked admin token.
  2. Close the IDOR class on read endpoints.Supervisor, DailyRouteAssignments, Outlets, PurchaseOrders, NotBillings, Stock. Add an ownership/subordinate-membership check (scope from JWT) — ideally in a shared base-controller helper so it can't be forgotten on the next endpoint.
  3. Fix soft-delete globally.Add the IsActive && !IsDeleted global query filter to User (and audit every IgnoreQueryFilters()), make DELETE set IsActive=false too, and convert the email/username/code unique indexes to filtered (WHERE IsDeleted = false).
  4. Make stock concurrency correct, not incidental.Wrap StockTaking submit/adjust in one EF transaction whose connection holds the FOR UPDATE lock; gate adjust on IsAdjusted; treat the advisory/Redis lock as an optimization, not the integrity boundary. Add a TOCTOU guard (row-version or FOR UPDATE) on Billings status transitions.
  5. Standardize pagination + enforce idempotency on money/stock writes.Clamp page/pageSize once in a shared paging helper; add DB unique backstops (e.g. BillingNumber, rep+outlet+date) so a double-submit without the header can't duplicate.
  6. Wire up (or remove) the access-token revocation infra; tighten Product price precision & validation.Either enforce IsRevokedAsync via OnTokenValidated + call RevokeAsync on logout, or delete the dead service. Configure decimal(18,2) on Product price columns.

07Scope & method

How this assessment was produced, so its limits are clear.

What was covered

Every feature with a controller (26 features, 31 controllers) was individually audited by an independent reviewer, plus a dedicated pass on cross-cutting infrastructure (middleware, auth, EF/DbContext, observability, secrets) and a baseline pass on the established pattern + test suite.

Rubric applied per feature

Layering · async & cancellation · input validation · structured error handling · EF query quality (tracking, projection, N+1, pagination) · DTO usage · security & IDOR · soft-delete compliance · business-logic & concurrency correctness · CLAUDE.md / never-do.md adherence.

Read-only

No files were modified. Findings cite file:line from the working tree at review time. Findings were not cross-validated by a second scorer, so treat individual Low items as leads; the Critical/High items each have concrete code evidence.

Not in scope

No build/typecheck/test execution (CI handles that). No sfa_web or sfa_mobile review. Performance/load testing not performed — concurrency findings are from code reasoning, not observed races.

One caveat on severity

This is a single-company deployment (no multi-tenancy), which softens the IDOR findings from "cross-tenant data breach" to "cross-user / cross-territory exposure within the company." That is still a real authorization defect — a sales rep seeing another rep's full ledger, or a supervisor seeing a non-subordinate's numbers — but it is not external data leakage. Prioritize accordingly.
